04 — Preprocessing

Data Preprocessing & Feature Dropping

The MLP notebook performs explicit dropping of two correlated features — unlike the RBF notebook which relied on select_dtypes alone.

Step 1 — Ordinal Encoding (for correlation analysis only)

Two categorical columns were ordinal-encoded on a working copy X_df_encoded before computing the correlation matrix:

🔢
background_app_usage_level
Low=0 · Medium=1 · High=2  →  ordinal scale
📶
signal_strength_avg
Poor=0 · Moderate=1 · Good=2  →  ordinal scale
Note: This encoding was only used to compute the full correlation heatmap. The final training matrix is built with select_dtypes(include=[np.number]) on the original unencoded X_df, which implicitly excludes these two string columns along with Device_ID.

Step 2 — Explicit Column Drops (unique to MLP notebook)

After identifying multicollinear pairs, two features were explicitly dropped from the concatenated dataframe before training:

# Explicit drop of highly correlated features
features_to_drop = [
    'thermal_stress_index',           # r=0.952 with avg_battery_temp_celsius
    'avg_charging_cycles_per_week'      # r=0.945 with avg_screen_on_hours_per_day
]

df = df.drop(columns=features_to_drop)
🌡️
thermal_stress_index  → DROPPED
Correlation of 0.952 with avg_battery_temp_celsius. Near-perfect redundancy — retaining both would add noise without information gain.
⚡
avg_charging_cycles_per_week  → DROPPED
Correlation of 0.945 with avg_screen_on_hours_per_day. Charging frequency is largely a function of screen time — multicollinear, hence removed.
🔑
Device_ID  → DROPPED (both CSVs)
UUID identifier — carries zero predictive signal. Excluded before all analysis.
🏷️
recommended_action  → DROPPED
Categorical label for a classification task — not used in this regression model.
🔡
background_app_usage_level  → DROPPED (implicit)
Still a string in the original X_df — excluded by select_dtypes(include=[np.number]).
📡
signal_strength_avg  → DROPPED (implicit)
Same as above — string dtype excluded automatically.

06 — Architecture

MLP Architecture

A feedforward neural network with two hidden layers, ReLU activations, and Adam optimiser implemented via sklearn.neural_network.MLPRegressor.

Total trainable parameters: 416 + 528 + 17 = 961

Forward Pass

# Pseudocode — forward pass through the MLP

# Layer 1: Input → Hidden 1
h1 = relu(X_scaled @ W1 + b1)   # shape: (N, 32)

# Layer 2: Hidden 1 → Hidden 2
h2 = relu(h1 @ W2 + b2)         # shape: (N, 16)

# Output: Hidden 2 → Prediction (linear — no activation for regression)
ŷ  = h2 @ W3 + b3               # shape: (N, 1)

Activation Function — ReLU

ReLU — sparse, fast gradients. Avoids vanishing gradient in shallow networks.

Loss Function & Optimiser

Training minimises Mean Squared Error (MSE) via the Adam optimiser:

# Loss
MSE = (1/N) × Σ (yᵢ − ŷᵢ)²

# Adam update rule (simplified)
mₜ = β₁·mₜ₋₁ + (1−β₁)·∇L    # first moment (momentum)
vₜ = β₂·vₜ₋₁ + (1−β₂)·∇L²   # second moment (adaptive LR)
θₜ = θₜ₋₁ − η · m̂ₜ / (√v̂ₜ + ε)

# sklearn defaults used
β₁=0.9, β₂=0.999, ε=1e-8, η=0.001 (initial)

10 — Evaluation

Evaluation Plots

The notebook generates five diagnostic visualisations to assess model quality.

📈 Predicted vs Actual
Scatter plot of ŷ vs y on the test set. Points clustered tightly around the red diagonal (y=x) confirm low systematic error.
〰️ Residual Plot
Residuals (y−ŷ) vs predicted values. Should be randomly scattered around zero — patterns here indicate model bias or heteroscedasticity.
📊 Residual Distribution
Histogram of (y−ŷ). Should be approximately normal, centred at zero. Skew indicates systematic over/under-prediction.
🟠 Absolute Error Distribution
Histogram of |y−ŷ|. Mean marked with a red dashed line. Right tail shows largest prediction errors.
📉 Training Loss Curve
MSE loss per epoch during training (2083.3 → 4.77). Shows convergence behaviour and where early stopping triggered.
〜 Smoothed Line Plot
Expected vs predicted battery health as smoothed lines (25-sample moving average) over sorted test samples — shows tracking quality across the health range.
11 — Usage

How to Run

# 1. Place both CSVs in the same directory as the notebook
#    smartphone_battery_features.csv
#    smartphone_battery_targets.csv

# 2. Install dependencies
pip install pandas numpy matplotlib seaborn scikit-learn

# 3. Open and run the notebook top-to-bottom
jupyter notebook MLP_Model.ipynb
Note: The MLP trains quickly (~5–10 seconds on CPU for 325 epochs). No GPU required — scikit-learn's MLPRegressor runs on CPU only. Run all cells in order; the early stopping will reproduce the 325-epoch result with random_state=42.
